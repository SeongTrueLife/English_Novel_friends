# Plan v2 — SentenceMate Reader 개선

작성: 2026-05-24
최종 업데이트: 2026-05-25 (시스템 프롬프트 확정, 단어장 모델 확정, 영어판 추가)

---

## Context

현재 앱을 사용하면서 사용자가 느낀 두 가지 핵심 한계:

1. **단어장 구조 문제** — AI 응답에서 단어와 설명을 함께 드래그해서 한 덩어리로 저장하는 방식이라 "외울 단어"와 "설명"이 구분되지 않음. 책 원문 문장이 앵커로 같이 저장되지 않아 암기 효율도 떨어짐.
2. **AI 프롬프트(3단계 구조)의 한계** —
   - 1단계 직독직해가 추론력을 키우기보다 한국어 매핑에 의존하게 만듦
   - 3단계가 가장 효과적이지만 "해당될 때만 선별" 규칙으로 사용자가 궁금한 단어 누락
   - 영어 본질에 가까운 문법 설명(시제·관사·전치사 등) 부족

**목적**: 추론력을 살리는 학습 사이클로 응답 구조를 재설계하고, 단어장을 단어/설명/예문(원문) 분리된 카드 형태로 만들며, AI 응답에서 단어 카드 + 영어식 사고 노트가 자동 생성 가능하도록 한다.

---

## 결정 사항

### A. 응답 구조 (프롬프트 재설계) — 확정

- **직독직해(구 1단계) 완전 제거** (추론력 죽이는 원인)
- **새 구조 3축** (JSON 응답):
  - **vocab**: 선택 문장의 단어 카드 후보. 한 줄 meaning에 핵심 이미지 노출
  - **thinking**: 영어식 사고 항목 (4 type: `core_image` / `grammar` / `culture` / `author_intent`)
  - **naturalTranslation**: 자연 해석 — UI에서 기본 접힘 (추론 사이클 보호)
- **페르소나**: "한국인 학습자 직관 잘 아는 bilingual 친구 + 인지언어학·SLA 배경"
- **톤**: 친구 반말 (~이야, ~거든)
- **노티싱**: 부드러운 방식 ("보통 X로 알지만 여기선 Y야...")
- **grammar 항목** 끝에 한 줄 **해석 가이드** 의무 ("그래서 'A is why B' = '...' 로 읽으면 돼")
- **단어 누락 안전망**: vocab 우선순위 기반 (빠짐없이 강제 X) + 사용자 추가 프롬프트가 보조 안전망
- **숙어/phrasal verb**: vocab 한 항목 + 분해 hint, 구성 단어 중복 금지
- **단독 전치사**: vocab 제외, thinking의 core_image/grammar에서 이미지 스키마로 다룸

### B. 출력 형식 — JSON Structured Output

- `responseMimeType: "application/json"` + `responseSchema` 강제 + 시스템 프롬프트 형식 명시 (이중 보장)
- 스트리밍 포기 trade-off 수용 (응답 완료까지 1~3초 빈 상태)
- 확정 스키마:

```js
{
  type: "object",
  properties: {
    vocab: {
      type: "array",
      items: {
        type: "object",
        properties: {
          word: { type: "string" },
          meaning: { type: "string" }
        },
        required: ["word", "meaning"]
      }
    },
    thinking: {
      type: "array",
      items: {
        type: "object",
        properties: {
          type: { type: "string", enum: ["core_image", "grammar", "culture", "author_intent"] },
          title: { type: "string" },
          body: { type: "string" }
        },
        required: ["type", "title", "body"]
      }
    },
    naturalTranslation: { type: "string" }
  },
  required: ["vocab", "thinking", "naturalTranslation"]
}
```

### B'. 시스템 프롬프트 언어 — 영어 본문, 한국어 출력

- 시스템 프롬프트 본문 = **영어** (LLM 이해도·토큰 효율)
- 출력값(vocab.meaning / thinking.body / naturalTranslation) = **한국어**
- 마커 = **영어 통일** (`[Book Info]`, `[Previous 1]`, `[Previous 2]`, `[Selected]`, `[Next 1]`, `[Next 2]`, `[User Request]`)
- 톤 지시 = 영어 본문 + 한국어 어미 인용 ("Use plain speech endings ~이야, ~거든")
- Few-shot 예시 입출력 = 한국어 그대로 (출력 언어 학습)
- **최종 영어 본문**: [system_prompt_v2_final_en.md](system_prompt_v2_final_en.md)
- 한국어 검토본(토론 기록): [system_prompt_v2_draft2.md](system_prompt_v2_draft2.md)

### C. 사용자 추가 프롬프트 — UI는 B+C 결합

- 시스템 프롬프트는 항상 전송. 사용자가 입력하면 `buildUserMessage`에 `[User Request]` 한 줄 추가
- 입력 UI:
  - **(B)** Ask AI 옆 `+` 토글 → 평소 즉시 전송, `+` 누르면 textarea(2~3줄 자동 확장) 펼침
  - **(C)** 응답 시트 안 follow-up 입력란 → **채팅형 누적(C-1)**
- follow-up은 multi-turn (Gemini `contents` 배열에 이전 대화 누적)
- 다른 문장 선택 시 새 세션 시작 (토큰 폭증 방지, 3~5턴 이내 예상)
- 프리셋 칩은 MVP 제외 (실사용 데이터 모인 후 추후 추가)
- 사용자 입력 내역 누적 (`userPromptHistory`) — 추후 패턴 분석용

### D. 단어장 (Wordbook) — kind 통합 모델

- **vocab + thinking 둘 다 단어장에 저장**. type/title 필드로 분류·필터링
- 데이터 모델:

```js
// 통합 컬렉션, kind로 단어 카드 vs 노트 카드 구분
[
  {
    id,
    kind: "word",
    word,
    meaning,
    exampleSentence, // 책 원문 (앵커)
    bookTitle,
    author,
    chapter,
    createdAt,
    // SRS (추후)
    reviewCount,
    lastReviewedAt,
    nextReviewAt,
    easeFactor,
  },

  {
    id,
    kind: "note",
    noteType: "core_image" | "grammar" | "culture" | "author_intent",
    title,
    body,
    exampleSentence,
    bookTitle,
    author,
    chapter,
    createdAt,
    reviewCount,
    lastReviewedAt,
    nextReviewAt,
    easeFactor,
  },
];
```

- 카드 추가 UX:
  - **vocab 카드**: 각 항목 옆 `+` 버튼 (가볍게)
  - **note 카드**: 각 thinking 항목 우상단 `📌 노트로 저장` 버튼 (더 의식적인 저장)
  - 둘 다 책 원문 문장(exampleSentence) 자동 첨부
- 복습 카드 UI (추후 SRS 단계):
  - **word**: 앞면=word, 뒷면=meaning + exampleSentence
  - **note**: 앞면=title + noteType 배지, 뒷면=body + exampleSentence
- 단어장 화면 탭 구조: "단어 / 노트" 분리, 노트 안에서 noteType 필터
- **기존 sm_vocab 데이터 마이그레이션 정책**: 별도 결정 필요 (다음 세션)

### E. 모델 — gemini-3-flash 유지

- 가성비 + JSON Structured Output 지원
- 가격 인상된 gemini-3.5-flash는 보류

### F. 디바이스 우선순위

- 1차: **태블릿**, 2차: **스마트폰**
- 모든 UI 결정은 이 기준 — 한 손 조작, 키보드 올라옴 대응

### G. 책 메타데이터 (책 정보 활용)

- 사용자가 책 추가 시 **책 제목 + 작가 이름** 직접 입력 (EPUB 자동 파싱은 보류)
- 메타데이터는 LocalStorage에 책 단위로 저장
- 모든 AI 호출 시 user message 첫 줄에 `[Book Info] {title} — {author}` 자동 첨부
- 시스템 프롬프트가 이걸 활용해 시대/문화/작가 문체 context 적용
- 모르는 책은 환각 금지 (일반 설명으로 fallback)
- 무명 책은 v2 범위 밖 (사용자 타겟이 영어 입문~중급이라 마이너 책 거의 안 봄)

### H. 맥락 문장 — 앞뒤 2문장씩

- 현재 useTextSelection.js에서 1문장씩 추출 → **2문장씩**으로 변경
- buildUserMessage에서 `[Previous 1]`, `[Previous 2]`, `[Next 1]`, `[Next 2]` 형식으로 join
- 균형: 1문장은 부족(대명사 가리킴 놓침), 3문장은 토큰 증가·집중 분산

## 검증 방법 (구현 시점)

- 태블릿 / 폰 브라우저에서 dev 서버로 실제 사용 흐름 테스트
- 다양한 난이도의 영어 문장으로 응답 품질 확인 (단어 누락, 자연 해석, 영어식 사고 깊이, grammar 해석 가이드 등)
- 단어 카드 + 노트 카드 추가 → 단어장 정상 표시 확인
- follow-up 채팅형 누적 토큰 사용량 모니터링 (3~5턴 이내 합리적인지)
- 책 메타데이터 활용 — 유명 책(예: Pride and Prejudice)에서 culture 항목이 시대 context 잘 반영하는지
- JSON Structured Output 형식 오류 발생률 (0%에 가까워야 정상)
