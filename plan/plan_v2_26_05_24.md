## Context

현재 앱을 사용하면서 사용자가 느낀 두 가지 핵심 한계:

1. **단어장 구조 문제** — AI 응답에서 단어와 설명을 함께 드래그해서 한 덩어리로 저장하는 방식이라 "외울 단어"와 "설명"이 구분되지 않음. 책 원문 문장이 앵커로 같이 저장되지 않아 암기 효율도 떨어짐.
2. **AI 프롬프트(3단계 구조)의 한계** —
   - 1단계 직독직해가 추론력을 키우기보다 한국어 매핑에 의존하게 만듦
   - 3단계가 가장 효과적이지만 "해당될 때만 선별" 규칙으로 사용자가 궁금한 단어 누락
   - 영어 본질에 가까운 문법 설명(시제·관사·전치사 등) 부족

**목적**: 추론력을 살리는 학습 사이클로 응답 구조를 재설계하고, 단어장을 단어/설명/예문(원문) 분리된 카드 형태로 만들며, AI 응답에서 단어 카드 + 영어식 사고 노트가 자동 생성 가능하도록 한다. 또한 소규모 배포(다수 사용자)까지 염두에 두고 클라우드 DB 기반으로 처음부터 설계한다.

## 결정 사항

### A. 응답 구조 (프롬프트 재설계) — 확정

- **직독직해(구 1단계) 완전 제거** (추론력 죽이는 원인)
- **새 구조 3축** (JSON 응답):
  - **vocab**: 선택 문장의 단어 카드 후보. 한 줄 meaning에 핵심 이미지 노출
  - **thinking**: 영어식 사고 항목 (4 type: `core_image` / `grammar` / `culture` / `author_intent`)
  - **naturalTranslation**: 자연 해석 — UI에서 기본 접힘 (추론 사이클 보호)
- **페르소나**: "한국인 학습자 직관 잘 아는 bilingual 친구 + 인지언어학·SLA 배경"
- **톤**: 친구 반말 (~이야, ~거든)
- **노티싱**: 부드러운 방식 ("보통 X로 알지만 여기선 Y야...")
- **grammar 항목** 끝에 한 줄 **해석 가이드** 의무 ("그래서 'A is why B' = '...' 로 읽으면 돼")
- **단어 누락 안전망**: vocab 우선순위 기반 (빠짐없이 강제 X) + 사용자 추가 프롬프트가 보조 안전망
- **숙어/phrasal verb**: vocab 한 항목 + 분해 hint, 구성 단어 중복 금지
- **단독 전치사**: vocab 제외, thinking의 core_image/grammar에서 이미지 스키마로 다룸

### B. 출력 형식 — JSON Structured Output

- `responseMimeType: "application/json"` + `responseSchema` 강제 + 시스템 프롬프트 형식 명시 (이중 보장)
- 스트리밍 포기 trade-off 수용 (응답 완료까지 1~3초 빈 상태)
- 확정 스키마:

```js
{
  type: "object",
  properties: {
    vocab: {
      type: "array",
      items: {
        type: "object",
        properties: {
          word: { type: "string" },
          meaning: { type: "string" }
        },
        required: ["word", "meaning"]
      }
    },
    thinking: {
      type: "array",
      items: {
        type: "object",
        properties: {
          type: { type: "string", enum: ["core_image", "grammar", "culture", "author_intent"] },
          title: { type: "string" },
          body: { type: "string" }
        },
        required: ["type", "title", "body"]
      }
    },
    naturalTranslation: { type: "string" }
  },
  required: ["vocab", "thinking", "naturalTranslation"]
}
```

### B'. 시스템 프롬프트 언어 — 영어 본문, 한국어 출력

- 시스템 프롬프트 본문 = **영어** (LLM 이해도·토큰 효율)
- 출력값(vocab.meaning / thinking.body / naturalTranslation) = **한국어**
- 마커 = **영어 통일** (`[Book Info]`, `[Previous 1]`, `[Previous 2]`, `[Selected]`, `[Next 1]`, `[Next 2]`, `[User Request]`)
- 톤 지시 = 영어 본문 + 한국어 어미 인용 ("Use plain speech endings ~이야, ~거든")
- Few-shot 예시 입출력 = 한국어 그대로 (출력 언어 학습)
- **최종 영어 본문**: [system_prompt_v2_final_en.md](../SystemPrompt/system_prompt_v2_final_en.md)
- 한국어 검토본(토론 기록): [system_prompt_v2_draft2.md](../SystemPrompt/system_prompt_v2_draft2.md)

### C. 사용자 추가 프롬프트 — UI는 B+C 결합

- 시스템 프롬프트는 항상 전송. 사용자가 입력하면 `buildUserMessage`에 `[User Request]` 한 줄 추가
- 입력 UI:
  - **(B)** Ask AI 옆 `+` 토글 → 평소 즉시 전송, `+` 누르면 textarea(2~3줄 자동 확장) 펼침
  - **(C)** 응답 시트 안 follow-up 입력란 → **채팅형 누적(C-1)**
- follow-up은 multi-turn (Gemini `contents` 배열에 이전 대화 누적)
- 다른 문장 선택 시 새 세션 시작 (토큰 폭증 방지, 3~5턴 이내 예상)
- 프리셋 칩은 MVP 제외 (실사용 데이터 모인 후 추후 추가)
- 사용자 입력 내역 누적 (`userPromptHistory`) — 추후 패턴 분석용

### D. 단어장 (Wordbook) — kind 통합 모델

- **vocab + thinking 둘 다 단어장에 저장**. kind 필드로 분류·필터링
- 데이터 모델 (개념 — Supabase 테이블 스키마는 다음 세션에서 확정):

```js
// 통합 컬렉션, kind로 단어/노트/문장 구분
[
  {
    id,
    userId,
    kind: "word",
    word,
    meaning,
    exampleSentence, // 책 원문 (앵커)
    bookId,
    chapter,
    createdAt,
    // SRS (v2엔 데이터 모델만, UI는 v3)
    reviewCount,
    lastReviewedAt,
    nextReviewAt,
    easeFactor,
    intervalDays,
  },

  {
    id,
    userId,
    kind: "note",
    noteType: "core_image" | "grammar" | "culture" | "author_intent",
    title,
    body,
    exampleSentence,
    bookId,
    chapter,
    createdAt,
    reviewCount,
    lastReviewedAt,
    nextReviewAt,
    easeFactor,
    intervalDays,
  },

  {
    id,
    userId,
    kind: "sentence", // v2엔 데이터 모델만, UI는 v3
    sentence, // 인상 깊은 문장 자체 (AI 응답 안 거치고 책 본문에서 직접 저장)
    note, // 사용자 메모 (선택)
    bookId,
    chapter,
    createdAt,
  },
];
```

- 카드 추가 UX:
  - **vocab 카드**: 각 항목 옆 `+` 버튼 (가볍게)
  - **note 카드**: 각 thinking 항목 우상단 `📌 노트로 저장` 버튼 (더 의식적인 저장)
  - **sentence 카드**: v3에서 UI 추가 예정. 책 본문 선택 시 "Ask AI" 옆에 "저장" 옵션
  - 모두 책 원문 + 챕터 정보 자동 첨부
- 복습 카드 UI (v3 SRS 단계):
  - **word**: 앞면=word, 뒷면=meaning + exampleSentence
  - **note**: 앞면=title + noteType 배지, 뒷면=body + exampleSentence
- 단어장 화면 탭 구조 (v2): "단어 / 노트" 분리, 노트 안에서 noteType 필터
- **기존 sm_vocab 데이터**: 폐기 결정 (사용자 단어장 비어있음, 마이그레이션 불필요)

### D'. 학습자료 범위 — v2 합의

학습자료를 어디까지 그릴지 v2 범위 합의:

| 범위                          | 내용                                                                           | v2 처리                                                                   |
| ----------------------------- | ------------------------------------------------------------------------------ | ------------------------------------------------------------------------- |
| **A** 단어장 + 노트           | 평면 리스트 + 책 필터 + 시간순/단어순                                          | ✅ UI까지 포함                                                            |
| **B** 책 단위 학습 카드       | 책별 단어/노트 모음 + 학습 통계 (책당 누적 단어/노트 수, 읽은 시간, 챕터 진척) | ✅ UI까지 포함 (챕터 정보는 이미 EpubReader가 TOC navigation으로 캡처 중) |
| **C** SRS (Spaced Repetition) | 망각 곡선 기반 복습 시점 자동 계산. SM-2 알고리즘                              | ⏳ 데이터 모델만 v2, UI는 v3                                              |
| **D** 사용자 컬렉션/태그      | 책 경계 넘어 사용자가 묶음 ("면접용 비즈니스" 등)                              | ❌ v2 제외 (사용자 데이터 쌓인 후 도입 자연스러움)                        |
| **E** 문장 컬렉션             | 인상 깊은 문장 자체 보존 (AI 응답 안 거침)                                     | ⏳ 데이터 모델만 v2 (kind: "sentence"), UI는 v3                           |

이유 요약:

- B는 책 메타데이터·챕터 정보가 어차피 있으니 거의 공짜로 따라옴 + 학습 동기 강화
- C/E는 마이그레이션 부담 없이 모델만 미리 깔아두면 v3 UI 추가가 빨라짐
- D는 빈 컬렉션 만들기 시작하면 학습 자체보다 정리에 시간 씀 → 사용자 데이터 어느 정도 쌓인 후 자연스럽게 도입

### E. 모델 — gemini-3-flash 유지

- 가성비 + JSON Structured Output 지원
- 가격 인상된 gemini-3.5-flash는 보류

### F. 디바이스 우선순위

- 1차: **태블릿**, 2차: **스마트폰**
- 모든 UI 결정은 이 기준 — 한 손 조작, 키보드 올라옴 대응

### G. 책 메타데이터 (책 정보 활용)

- 사용자가 책 추가 시 **책 제목 + 작가 이름** 직접 입력 (EPUB 자동 파싱은 보류)
- 메타데이터는 **Supabase `books` 테이블**에 저장 (사용자별, book_hash로 같은 책 매칭)
- 모든 AI 호출 시 user message 첫 줄에 `[Book Info] {title} — {author}` 자동 첨부
- 시스템 프롬프트가 이걸 활용해 시대/문화/작가 문체 context 적용
- 모르는 책은 환각 금지 (일반 설명으로 fallback)
- 무명 책은 v2 범위 밖 (사용자 타겟이 영어 입문~중급이라 마이너 책 거의 안 봄)

### H. 맥락 문장 — 앞뒤 2문장씩

- 현재 useTextSelection.js에서 1문장씩 추출 → **2문장씩**으로 변경
- buildUserMessage에서 `[Previous 1]`, `[Previous 2]`, `[Next 1]`, `[Next 2]` 형식으로 join
- 균형: 1문장은 부족(대명사 가리킴 놓침), 3문장은 토큰 증가·집중 분산

### I. 백엔드 & DB — Supabase + 익명 인증 (옵션 D 채택)

- **DB**: Supabase (PostgreSQL 기반 BaaS) — 관계형 구조, RLS, 익명 인증, Realtime, Storage 다 포함
- **인증**: **익명 인증(Anonymous Auth)으로 시작 → 나중에 이메일/Google OAuth 연결**
  - 앱 첫 진입 시 자동으로 anonymous user 생성 → 데이터는 처음부터 user_id 묶여서 클라우드에 쌓임
  - "데이터 보존" 동기로 사용자에게 이메일 연결 prompt 부드럽게 노출 (시점 추후 결정)
  - 동일 user_id 유지된 채 정식 계정 업그레이드 (데이터 마이그레이션 불필요)
- **인덱스 + RLS는 처음부터 설계** — 100명 → 1000명 가도 코드 안 바뀌게
  - RLS = Row Level Security. user_id로 본인 데이터만 접근 가능하게 PostgreSQL 정책으로 강제
- **비용 예상 (100명 규모)**: DB ~170MB / Storage ~300MB / Bandwidth 2~3GB → Free tier (500MB DB / 1GB Storage / 5GB egress) 안에 충분
- **vendor lock-in 낮음**: 오픈소스, 셀프호스팅 옵션 있음

### J. 스토리지 정책 — 하이브리드

저작권 + 비용 + 사용자 편의 균형:

| 데이터 종류                                   | 저장 위치               | 이유                                                            |
| --------------------------------------------- | ----------------------- | --------------------------------------------------------------- |
| 저작권 free 책 (앱 기본 제공 50~100권)        | Supabase Storage        | 우리가 합법적으로 호스팅 가능, 모든 사용자 공유                 |
| 사용자 업로드 epub 파일                       | IndexedDB (단말기 로컬) | 저작권 책임 회피 ("본인 단말기에 본인 파일 둠"), Storage 부담 0 |
| 책 메타데이터 (title·author·book_hash·source) | Supabase DB             | 단말기 간 sync, 같은 책 자동 매칭                               |
| 단어장·노트·읽기 진행·학습 통계               | Supabase DB             | sync가 핵심 가치                                                |

**book_hash 트릭**:

- 사용자가 epub 업로드 시 SHA-256 해시 계산
- 같은 책인지 식별 키 (제목·작가는 사용자 입력 다를 수 있어 불안정)
- 폰에서 같은 epub 올리면 hash 일치 → 기존 단어장/진행상황 자동 연결
- UX: "이 책 전에 태블릿에서 읽으셨네요. 이어서 보기?" 같은 prompt

**웹앱 한계 짚어둠**:

- 웹앱은 브라우저 샌드박스에 갇혀 OS 파일 시스템 자유 접근 불가 → IndexedDB가 유일한 현실적 답
- 단점: 단말기마다 epub 파일 한 번씩 올려야 함 (사용자 본인이 어차피 가진 파일이라 큰 부담 X)

### K. 배포 형태 — 웹앱 유지, 배포 직전 PWA 업그레이드

- **현재 웹앱 유지**: React + Vite + epub.js. 한 코드베이스로 모든 디바이스
- **PWA(Progressive Web App)로 살짝 업그레이드** (배포 직전):
  - `vite-plugin-pwa` 추가 → "홈 화면에 추가" 가능, 오프라인 작동, 브라우저 푸시 알림
  - 거의 공짜로 적용. SRS 복습 알림(v3) 같은 거 자연스럽게 됨
- **네이티브 앱(React Native/Flutter) 전환은 불필요**:
  - 빠른 개발/배포 손해, 플랫폼별 코드 부담
  - epub.js 기반 잘 동작 — 다시 짜면 epub 라이브러리 함정 재발견
  - 사용자 10,000명+ 시점에나 고민할 일

### L. 코드 마이그레이션 정책 — in-place 교체

- **새 폴더 안 만들고 현재 sentencemate-reader 디렉토리에서 그대로 교체**
- 기존 코드 평가:
  - **유지 (~30%)**: useTextSelection.js, EpubReader.jsx의 epub.js 초기화·테마 주입·paginated scroll 보정, BottomSheet, AskAIButton, 챕터 추적, CFI 읽기 진행 저장
  - **부분 수정 (~50%)**: useGeminiAPI(스트리밍→JSON), VocabList(새 모델), 책 추가 UX
  - **폐기 (~20%)**: AIResponse의 parseStages, storage.js의 LocalStorage 단어장 로직, systemPrompt.js 한국어 본문
- 이유:
  - epub.js 같이 함정 많은 라이브러리 위에 잘 잡혀있는 토대 다시 깔면 옛날 버그 재발견 위험
  - "기능 덧붙이기"가 아니라 "기능 통째로 교체" 케이스라 손볼 데가 명확하고 좁음
- 예외: React 구조 자체(라우팅 도입, Zustand 같은 상태관리 도입 등)를 크게 바꾸고 싶으면 그때 새 폴더 고민

## 영향 받는 파일 (구현 시 변경 영역)

- [systemPrompt.js](../sentencemate-reader/src/utils/systemPrompt.js) — `SYSTEM_PROMPT` 영어 본문으로 교체, `buildUserMessage` 영어 마커 + Book Info / User Request / 앞뒤 2문장 슬롯
- [useGeminiAPI.js](../sentencemate-reader/src/hooks/useGeminiAPI.js) — JSON Structured Output 모드 (`responseMimeType` + `responseSchema`), multi-turn 지원 (`contents` 배열 누적), 스트리밍 제거
- [useTextSelection.js](../sentencemate-reader/src/hooks/useTextSelection.js) — 앞/뒤 문장 1개 → 2개씩 추출
- [AIResponse.jsx](../sentencemate-reader/src/components/AIResponse.jsx) — `parseStages` 제거. JSON 응답 → vocab 칩 / thinking 아코디언 / naturalTranslation(접힘) 컴포넌트 분기. 카드 추가 UI (vocab `+`, note `📌`). follow-up 채팅형 누적 UI
- [storage.js](../sentencemate-reader/src/utils/storage.js) — 대부분 폐기. Supabase 클라이언트 호출 어댑터로 교체. IndexedDB 어댑터(epub 파일 로컬 저장)도 같이 둠
- [EpubReader.jsx](../sentencemate-reader/src/components/EpubReader.jsx) — Ask AI 옆 `+` 토글, 사용자 추가 프롬프트 상태 관리, 책 메타데이터 전달
- [VocabList.jsx](../sentencemate-reader/src/components/VocabList.jsx) — 새 모델 표시 (단어 탭 / 노트 탭 분리, noteType 필터, 책별 그룹)
- [FileUploader.jsx](../sentencemate-reader/src/components/FileUploader.jsx) — 업로드 시 메타데이터(제목·작가) 입력 UI 추가, book_hash 계산 후 Supabase에 메타데이터 저장 + IndexedDB에 파일 저장
- **신규 파일**:
  - `utils/supabase.js` — Supabase 클라이언트 초기화 + 익명 인증 자동 처리
  - `utils/indexeddb.js` — epub 파일 IndexedDB 어댑터 (저장·로드·삭제)
  - `utils/bookHash.js` — SHA-256 해시 계산
  - `components/BookLibrary.jsx` — 다중 책 라이브러리 화면 (현재는 단일 책 로드 형태)
  - `components/BookMetaForm.jsx` — 책 추가 시 메타데이터 입력 모달

## 검증 방법 (구현 시점)

- 태블릿 / 폰 브라우저에서 dev 서버로 실제 사용 흐름 테스트
- 다양한 난이도의 영어 문장으로 응답 품질 확인 (단어 누락, 자연 해석, 영어식 사고 깊이, grammar 해석 가이드 등)
- 단어 카드 + 노트 카드 추가 → 단어장 정상 표시 확인 (Supabase에 쌓이는지)
- 단말기 간 sync 확인 — 태블릿에서 추가한 단어장이 폰에서 보이는지
- book_hash 매칭 확인 — 같은 epub 다른 단말기서 올렸을 때 단어장 자동 연결되는지
- follow-up 채팅형 누적 토큰 사용량 모니터링 (3~5턴 이내 합리적인지)
- 책 메타데이터 활용 — 유명 책(예: Pride and Prejudice)에서 culture 항목이 시대 context 잘 반영하는지
- JSON Structured Output 형식 오류 발생률 (0%에 가까워야 정상)
- RLS 정책 동작 확인 — 사용자 A가 사용자 B의 단어장 못 보는지
- 익명 user → 이메일 연결 흐름 — 데이터 그대로 이어지는지
