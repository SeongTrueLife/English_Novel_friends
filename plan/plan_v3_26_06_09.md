## Context

현재 앱(v1)을 쓰며 느낀 두 핵심 한계 — plan_v2와 동일:

1. **단어장 구조 문제** — AI 응답에서 단어와 설명을 함께 드래그해 한 덩어리로 저장 → "외울 단어"와 "설명"이 안 나뉘고, 책 원문 앵커가 같이 안 붙어 암기 효율 저하.
2. **AI 프롬프트(구 3단계) 한계** — 직독직해가 추론력보다 한국어 매핑 의존을 키움 / 선별 규칙으로 궁금한 단어 누락 / 영어 본질 문법 설명 부족.

**목적**: 추론력을 살리는 학습 사이클로 응답을 재설계하고, 단어장을 **단어·문법·예문(원문) 분리 카드**로 만들며, AI 응답에서 단어 카드 + 영어식 사고 노트가 자동 생성되게 한다. 소규모 배포(다수 사용자)까지 염두에 두고 **클라우드 DB(Supabase) 기반**으로 처음부터 설계한다.

## 결정 사항

### A. 응답 구조 (프롬프트 v3) — 확정

직독직해(구 1단계) 완전 제거. 응답은 **학습 단위 중심 4축 JSON**:

- **vocab** — 단어/숙어 1개 = 학습 단위 1개. `word` + `meaning`(핵심 이미지 노출) + 그 단어에 종속된 `thinking` 자식 배열(type: `core_image`/`culture`/`author_intent`)
- **grammar** — 문법 패턴 1개 = 학습 단위 1개. `pattern` + `explanation` + `interpretation_guide`(한 줄 해석 가이드 의무)
- **sentence_thinking** — 어느 단어에도 안 매달리는 문장 차원 코멘트(type: `culture`/`author_intent`, 드묾)
- **naturalTranslation** — 자연 해석(UI 기본 접힘, 추론 사이클 보호)

페르소나·톤·노티싱·숙어/전치사 처리 등 상세 규칙과 few-shot은 **정본 [system_prompt_v3_final_en.md](../SystemPrompt/system_prompt_v3_final_en.md) 참조** (여기 복붙 안 함).

> v2 대비: thinking 평면 배열 → vocab/grammar/sentence_thinking 학습 단위로 재편. grammar 승격. note type 폐기.

### B. 출력 형식 — JSON Structured Output

- `responseMimeType: "application/json"` + `responseSchema` 강제 + 시스템 프롬프트 형식 명시 (이중 보장)
- 스트리밍 포기 trade-off 수용(응답 완료까지 1~3초)
- **확정 스키마(키·enum)는 system_prompt_v3 문서 기준** — vocab/grammar/sentence_thinking/naturalTranslation 구조. `example_sentence`는 응답에 포함 안 함(클라이언트가 `[Selected]` 자동 첨부).

### B'. 시스템 프롬프트 언어 — 영어 본문, 한국어 출력 (유지)

- 본문 = 영어(LLM 이해도·토큰 효율), 출력값(meaning/body/explanation/interpretation_guide/naturalTranslation) = 한국어
- 마커 = 영어 통일(`[Book Info]`, `[Previous 1/2]`, `[Selected]`, `[Next 1/2]`, `[User Request]`)
- 톤 지시 = 영어 본문 + 한국어 어미 인용, few-shot 입출력 = 한국어 그대로

### C. 사용자 추가 프롬프트 — UI는 B+C 결합 (유지)

- 시스템 프롬프트는 항상 전송. 사용자 입력 시 `buildUserMessage`에 `[User Request]` 한 줄 추가
- 입력 UI: (B) Ask AI 옆 `+` 토글 → textarea 펼침 / (C) 응답 시트 안 follow-up 입력란 → **채팅형 누적**
- follow-up은 multi-turn(Gemini `contents` 누적). 다른 문장 선택 시 새 세션. 프리셋 칩은 제외.
- **`userPromptHistory`(입력 내역 누적)는 현재 빌드 제외** — 운영/분석용이라 다음 단계로. follow-up 멀티턴 대화도 **DB에 저장 안 함(휘발성)**.

### D. 학습 카드 모델 (구 "단어장") — cards/sentences 분리, note 폐기

v3 응답 구조에 맞춰 재편. **상세 스키마는 [db_schema_v2.md](../db/db_schema_v2.md) 정본 참조**, 여기선 개념만:

- **`cards`** — 학습 단위(SRS 대상). `kind: word | grammar` 통합 모델.
  - word: `word` + `meaning` + `thinking`(JSONB, 그 단어 종속 인지/문화/의도)
  - grammar: `pattern` + `explanation` + `interpretation_guide`
  - 공통: `example_sentence`(책 원문 앵커, NOT NULL) + `chapter` + SRS 컬럼(`review_count`/`last_reviewed_at`/`next_review_at`/`ease_factor`/`interval_days`)
- **`sentences`** — 인상 깊은 문장 보관(학습 단위 아님). 데이터 모델은 현재 빌드, **UI는 다음 단계**.
- **note 폐기**: grammar는 독립 카드로 승격, core_image/culture/author_intent는 단어 카드의 thinking에 흡수 → 별도 note 카드 없음.
- **sentence_thinking / naturalTranslation은 저장 안 함**(표시만). sentence_thinking은 다음 단계에서 문장 저장 시 `sentences.note`로 흡수 검토.

카드 추가 UX:

- **vocab 카드**: 각 항목 옆 `+` 버튼(가볍게) — 단어+thinking 통째로 한 카드
- **grammar 카드**: 각 grammar 항목에 저장 버튼
- ~~note `📌` 버튼~~ — 폐기(note 개념 없음)
- **sentence 카드**: 다음 단계 UI. 책 본문 선택 시 "저장" 옵션
- 모두 책 원문(example_sentence) + 챕터 자동 첨부

단어장 화면 탭(현재 빌드): **"단어 / 문법"** 분리(책별 필터·정렬). 복습 카드 UI(SRS)는 다음 단계.

**기존 sm_vocab 데이터**: 폐기(사용자 단어장 비어있음, 마이그레이션 불필요).

### D'. 학습자료 범위 (갱신)

### G. 책 메타데이터 (유지 + books 테이블 확정)

- 책 추가 시 **EPUB에서 제목·작가·표지 자동 파싱해 채우고, 항상 수정 가능**(2026-06-11 프론트 설계에서 갱신 — 기존 "직접 입력/자동 파싱 보류"에서 변경). 제목 필수 / 작가 선택. 파싱 엉성하면 파일명에서 추측 + 경고, 표지 없으면 타이포 폴백. 상세: [frontend/frontend_plan_v1.md](frontend/frontend_plan_v1.md) §6.5
- **`books` 테이블에 저장**(글로벌 단일 행, `book_hash`로 같은 책 매칭)
- 모든 AI 호출 user message 첫 줄에 `[Book Info] {title} — {author}` 자동 첨부 → 시스템 프롬프트가 시대/문화/작가 문체 context 적용
- 모르는 책은 환각 금지(일반 설명 fallback). 무명 책은 범위 밖.

### H. 맥락 문장 — 앞뒤 2문장씩 (유지)

- useTextSelection.js: 1문장 → **2문장씩** 추출
- buildUserMessage에서 `[Previous 1/2]`, `[Next 1/2]`로 join (1문장 부족·3문장 과함의 균형)

### I. 백엔드 & DB — Supabase + 익명 인증 (유지 + 스키마 확정)

- **DB**: Supabase(PostgreSQL BaaS) — 관계형, RLS, 익명 인증, Realtime, Storage
- **인증**: 익명 인증 시작 → 추후 이메일/Google OAuth 연결(같은 user_id 유지, 데이터 마이그레이션 불필요)
- **사용자 테이블**: `auth.users` 직접 참조(별도 public.users 안 만듦, 필요해지면 profiles 추가)
- **인덱스 + RLS 처음부터 설계** — 100명 → 1000명 가도 코드 안 바뀌게
- **비용(100명)**: Free tier 안에 충분(DB ~170MB / Storage ~300MB / Egress 2~3GB)

### J. 스토리지 정책 — 하이브리드 (유지)

| 데이터                                       | 위치             | 이유                              |
| -------------------------------------------- | ---------------- | --------------------------------- |
| 저작권 free 책(기본 제공 50~100권)           | Supabase Storage | 합법 호스팅, 전 사용자 공유       |
| 사용자 업로드 epub 파일                      | IndexedDB(로컬)  | 저작권 책임 회피, Storage 부담 0  |
| 책 메타데이터(title·author·book_hash·source) | Supabase DB      | 단말기 간 sync, 같은 책 자동 매칭 |
| 단어장·문장·읽기 진행·세션                   | Supabase DB      | sync가 핵심 가치                  |

- **book_hash 트릭**: epub SHA-256 해시로 같은 책 식별 → 다른 단말기서 올려도 단어장/진행 자동 연결
- **웹앱 한계**: 브라우저 샌드박스라 IndexedDB가 유일한 로컬 파일 답. 단말기마다 epub 한 번씩 올림(본인 파일이라 부담 적음).

### K. 배포 형태 — 웹앱 유지, 배포 직전 PWA (유지)

- React + Vite + epub.js 한 코드베이스. 배포 직전 `vite-plugin-pwa`로 "홈 화면 추가"·오프라인·푸시(다음 단계 SRS 알림). 네이티브 전환 불필요(10,000명+ 시점 고민).

### L. 코드 마이그레이션 — **fresh scaffold** (2026-06-15 갱신: 기존 "in-place 교체"에서 변경)

> **변경 사유(2026-06-15)**: v2 아키텍처(`features/`·`services/`·`lib/` 구조, react-router, TanStack Query, Zustand, Supabase, Edge Function)가 v1엔 하나도 없어 사실상 전면 재작성이다. in-place로 뜯으면 v2를 짜는 AI가 옛 패턴(LocalStorage `storage.js`, 클라측 `systemPrompt.js`, `parseStages`)에 휩쓸릴 위험 → **빈 캔버스에 새 구조로 작성**이 더 깔끔하다고 판단.

- **fresh scaffold**: 기존 `sentencemate-reader/` → **`sentencemate-reader-v1/`로 rename해 읽기전용 참고로 동결**(배포 안 함). 원래 이름 `sentencemate-reader/`에 Vite로 v2를 새로 scaffold.
- **"재사용 30%"의 실체 = 스니펫 복사**: epub.js 초기화·테마·paginated 보정, BottomSheet, 챕터 추적, CFI 진행 로직, 문장 추출 — v1에서 **복사해 새 구조에 이식**(in-place 수정이 아님).
- **새로 작성(~50%)**: AI 호출(Edge Function 경유), VocabList(새 모델), 책 추가 UX
- **버림(~20%)**: parseStages, LocalStorage 단어장 로직, 한국어 시스템 프롬프트 본문
- **배포 충돌 없음**: 번들러는 진입점에서 import된 파일만 빌드 → v1 폴더는 v2가 import 안 하니 번들·런타임 무관. Vercel **Root Directory = `sentencemate-reader`(v2)**로 지정하면 v1은 빌드 대상에서 제외. v2 안정화 후 v1 폴더 삭제 가능(git 히스토리에 보존).

## M. DB 스키마 (확정 — 요약)

**정본은 [db_schema_v2.md](../db/db_schema_v2.md)** (테이블·컬럼·인덱스·RLS·시드 전체 SQL). 여기선 요약만.

### 테이블 (auth.users + public 6개 = ai_usage 포함)

| 테이블                         | 역할                                                                                     |
| ------------------------------ | ---------------------------------------------------------------------------------------- |
| **auth.users** (Supabase 관리) | 사용자(익명+이메일 연동). id/email/is_anonymous/created_at                               |
| **books**                      | 책 메타데이터(글로벌 단일 행, `book_hash` UNIQUE, `source`: user_upload/curated_free)    |
| **user_books**                 | users↔books N:M 중간 + 진척도(PK `(user_id, book_id)`, progress_cfi/pct, last_opened_at) |
| **cards**                      | 학습 카드(kind: word/grammar 통합, thinking JSONB, example_sentence, SRS 컬럼)           |
| **sentences**                  | 인상 깊은 문장 보관(데이터 모델만, UI는 다음 단계)                                       |
| **reading_sessions**           | 읽기 세션 풀 기록(started/ended/last_activity_at, start/end cfi·chapter)                 |
| **ai_usage** (v2.1)            | 일일 AI 호출 쿼터 카운터(`(user_id, usage_date)` PK). Edge Function이 service_role로만 접근 |

### 핵심 설계 결정

- **books 글로벌 단일 행** + user_books 중간 테이블 → 책 정보 중복 제거(3NF), book_hash 매칭·시드 공유 극대화
- **cards 통합 모델 + CHECK 제약** → kind별 컬럼 NULL/NOT NULL 강제(통합 모델 약점 보강)
- **thinking은 JSONB** → 그 단어와만 살고 검색 안 함, 같은 type 여러 개 가능
- **sentence_thinking / naturalTranslation 미저장**(표시만)
- **reading_sessions 풀 기록** + 미종료 세션 처리(클라 best-effort 마감 + 다음 세션 시작 시 정리 + `last_activity_at`)
- **SRS 컬럼**(word·grammar 둘 다 복습 대상) — `last_reviewed_at` 포함
- **`updated_at` + moddatetime 트리거**(cards/sentences/user_books) → 단말기 간 sync 충돌 기준 시각. reading_sessions는 last_activity_at이 대체
- **데이터 생명주기**: 라이브러리에서 책 제거(user_books DELETE) 시 cards/sentences **보존**. CASCADE는 사용자 계정 삭제 / 개별 카드 삭제 때만

### 인덱스 / RLS / 시드 (요약)

- **인덱스**: `(user_id, ...)` 패턴 **총 6개**(표준 4 + 부분 2 — 복습 일정 카드 / 미종료 세션이 부분). PK·UNIQUE 자동 인덱스는 제외. `ai_usage`는 PK가 조회를 커버해 별도 인덱스 없음.
- **RLS**: **6개 public 테이블** 전부 enable(books·user_books·cards·sentences·reading_sessions·ai_usage). 개인 데이터는 `user_id = auth.uid()`(모든 작업), books는 글로벌 SELECT + 인증 사용자 INSERT(`source='user_upload'`만). `ai_usage`는 사용자 정책 없이 **service_role 전용**. 운영자는 service_role bypass.
- **시드(저작권 free 책)**: 운영 스크립트(service_role) + Git 관리, `source='curated_free'`, book_hash UNIQUE로 idempotent. 책 선별·노출 UX는 다음 단계.
