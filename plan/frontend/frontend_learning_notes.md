## 강의 0: 큰 그림 — 프론트엔드 앱의 4개 질문

화면 있는 앱을 만들 때 뼈대를 정한다는 건 결국 이 4가지를 답하는 것:

| 질문 | 답하는 것 | 이 노트의 강의 |
| --- | --- | --- |
| **어떤 화면을 보여줄까?** | 라우팅 | 강의 2 |
| **데이터를 어디에 담고 어떻게 흘릴까?** | 상태관리 | 강의 3·4·5 |
| **파일을 어떻게 정리할까?** | 폴더 구조 | 강의 6 |
| **어떤 순서로 만들까?** | 구현 시퀀싱 | 강의 7 |

설계의 핵심 원칙 한 줄: **"바꾸기 비싼 것만 미리 못 박고, 싼 것은 만들며 다듬는다."** 라우팅·상태관리는 나중에 바꾸면 코드를 갈아엎어야 해서 먼저 정하고, 버튼 위치 같은 건 구현 중에 정한다.

## 강의 1: SPA와 렌더링 — 기본 어휘

### 렌더링 (rendering)

- **렌더링 = 상태(데이터)를 화면(픽셀)으로 그리는 것.**
- React의 핵심 사고방식: **화면은 상태의 거울이다.** 내가 직접 DOM을 만지지 않고, "상태"를 바꾸면 React가 알아서 화면을 다시 그린다.
- 공식 비유: `UI = f(state)`. 같은 상태면 같은 화면. → 그래서 "상태관리"가 프론트의 절반이다.

### SPA (Single Page Application)

전통 웹과 SPA의 차이:

```
[전통 웹 (MPA)]
주소 클릭 → 서버가 새 HTML 페이지를 통째로 보냄 → 브라우저 깜빡 → 새 화면
  (네이버 옛날 방식: 페이지마다 서버 왕복)

[SPA]
처음 한 번만 앱(JS 번들)을 받음 → 이후 화면 전환은 브라우저 안에서 JS가 처리
  → 서버 왕복 없음, 깜빡임 없음. 데이터만 필요할 때 따로 받아옴(API).
```

- 우리 앱(React+Vite)은 **SPA**. epub 읽다가 단어장 갔다가 통계 가도 페이지 리로드가 없다.
- 그래서 "URL이 바뀐다"의 의미가 전통 웹과 다르다 → 강의 2.

### 컴포넌트 / props / state (최소 어휘)

- **컴포넌트**: 화면 조각을 만드는 함수. `function BookCover() { return <div>...</div> }`
- **props**: 부모가 자식에게 내려주는 입력값(읽기 전용). 함수 인자 같은 것.
- **state**: 컴포넌트가 들고 있는, 바뀔 수 있는 값. 바뀌면 그 컴포넌트가 리렌더.
- **props drilling(프롭 내리꽂기)**: 값을 멀리 떨어진 자식에게 주려고 중간 컴포넌트들을 줄줄이 거치는 것. 이게 귀찮아서 "전역 상태 공유" 도구가 나온다(강의 5).

## 강의 2: 라우팅 — 어떤 화면을 보여줄지

### 라우팅 vs 라우터

- **라우팅(routing)** = "지금 어떤 화면을 보여줄지 결정하는 규칙/행위".
- **라우터(router)** = 그 규칙을 실행하는 도구. 하는 일은 **"현재 URL을 읽어 → 거기 매핑된 컴포넌트를 골라 렌더"**. URL이 입력, 화면이 출력.

### 두 학파: state에 적기 vs URL에 적기

"지금 어떤 화면인가"를 어디에 적어두느냐:

| | **state 라우팅** | **URL 라우팅** |
| --- | --- | --- |
| 어디에 | `useState('library')` | 브라우저 주소 `/library` |
| 화면 2개일 때 | 충분 | 과함 |
| 화면 7~8개 | 무너짐 | 적합 |
| 뒤로가기 버튼 | ❌ 직접 구현 | ✅ 공짜 |
| 새로고침 위치 보존 | ❌ (state 초기화) | ✅ |
| 딥링크 | ❌ | ✅ |

**핵심**: URL 라우팅이 "더 고급"이라서 쓰는 게 아니다. **뒤로가기·새로고침 보존·딥링크라는 *필요*가 생겼을 때** 쓰는 인프라다. 화면이 진짜 2개면 state가 정답.

### 클라이언트 사이드 라우팅 (중요 오해 방지)

SPA에서 URL이 바뀌어도 **페이지를 새로 받아오지 않는다.** react-router는 브라우저 주소만 바꾸고(History API) 화면 컴포넌트만 교체한다 → 깜빡임 없이 즉각적. 이게 "클라이언트 사이드 라우팅".

```
주소 /vocab 로 바뀜  →  서버 왕복 X  →  라우터가 <VocabList/> 를 그 자리에 렌더
```

### 라우트 vs 오버레이 (가장 헷갈리는 구분)

> **그 자체가 목적지(destination)면 라우트, 지금 맥락 위 곁일(sub-task)이면 오버레이.**

- **라우트(주소 바뀜)**: 서재·리더·단어장·통계 — 화면을 통째로 갈아끼움.
- **오버레이(주소 안 바뀜, 그냥 state)**: 모달·바텀시트·팝업 — 지금 화면 *위에* 잠깐 떠오르고 닫으면 원래 맥락 그대로.

왜 오버레이를 라우트로 만들면 안 되나: 리더에서 단어를 물어 AI 시트가 떴는데 주소까지 바뀌면, 뒤로가기를 눌렀을 때 "시트만 닫을지 / 리더를 나갈지"가 모호해진다. 오버레이는 *화면이 아니라 현재 화면의 한 상태*다.

판단 기준 표:

| 이게 떠 있는 동안 | 종류 |
| --- | --- |
| 다른 화면으로 완전히 이동한 느낌 | 라우트 |
| 원래 화면이 뒤에 그대로 있고 잠깐 덮은 느낌 | 오버레이 |

### 중첩 라우트 / 쿼리파라미터 (보충)

- **중첩 라우트(nested route)**: `/stats`(대시보드)와 `/stats/:bookId`(책별 상세)처럼 부모-자식 화면. 공통 레이아웃(셸)을 부모가 들고, 자식만 바뀐다.
- **경로 파라미터(`:bookId`)**: URL의 변하는 부분. `/read/abc123` → `bookId = "abc123"`. "어떤 책인가"를 URL이 들고 있어 새로고침해도 그 책으로 복귀.
- **쿼리파라미터(`?tab=word`)**: 같은 화면 안의 작은 상태(탭 선택 등)를 URL에 담을 때. 딥링크·새로고침에 탭 유지가 필요하면 쓰고, 아니면 그냥 컴포넌트 state로 둬도 됨.

## 강의 3: 상태의 두 종류 — 프론트 상태관리의 출발점

상태관리에서 헤매는 99%의 원인은 **두 종류의 상태를 한 통에 넣고 같은 도구로 다루려 해서**다. 먼저 칼같이 가른다.

### ① 서버 상태 (Server State / 서버 캐시)

- **진짜 주인이 원격 서버(DB)에 있는 데이터.** 내 화면의 것은 **복사본(캐시)**일 뿐.
- 예: 단어 카드 목록, 라이브러리 책 목록, AI 응답, 읽기 진척, 통계.
- 특징 (= 골칫거리의 근원):
  - **stale(낡음)**: 다른 기기에서 바꾸면 내 복사본은 옛것이 됨 → 다시 받아와야(refetch).
  - **비동기 + 네트워크**: 항상 **로딩 중 / 실패**가 따라다님.
  - **공유**: 같은 데이터가 여러 화면에 동시에 나옴(카드 → 단어장·통계·책별상세).

### ② 클라이언트 상태 (Client / UI State)

- **진짜 주인이 이 브라우저 안에만 있는 데이터.**
- 예: 다크모드, 폰트 크기, AI 시트 열림/닫힘, 단어장 현재 탭, 지금 선택한 문장.
- 특징: 네트워크 없음, 로딩/에러 개념 없음, 그냥 메모리 변수. 새로고침하면 사라져도 대개 OK.

### 분류 연습 (SentenceMate)

| 데이터 | 종류 | 도구 |
| --- | --- | --- |
| 카드 목록·라이브러리·AI응답·진척·통계 | 서버 | TanStack Query |
| 로그인 user 세션 | 클라(전역) | Zustand |
| 다크모드·폰트·읽기설정 | 클라(전역) | Zustand |
| 탭 선택·시트 열림·입력창 초안 | 클라(로컬) | useState |

> 대부분의 앱 데이터가 서버 상태다(특히 "DB를 보여주고 동기화하는 앱"). → **서버 상태를 잘 다루는 도구가 앱의 심장.**

### 함정: 서버 상태를 useState로 다루기

```jsx
// 라이브러리 없이 — 화면 하나면 OK, 여러 화면이면 지옥
const [cards, setCards] = useState([])
const [loading, setLoading] = useState(false)
const [error, setError] = useState(null)
useEffect(() => {
  setLoading(true)
  getCards().then(setCards).catch(setError).finally(() => setLoading(false))
}, [])
```

문제: 카드가 3개 화면에 나오면 → 각자 fetch, 각자 loading/error 변수, 카드 저장 시 3곳 수동 갱신. **이 보일러플레이트와 동기화 버그가 서버 상태의 진짜 비용.** 이걸 전문으로 없애는 도구가 다음 강의.

## 강의 4: TanStack Query — 서버 상태 전용 도구

### 무엇을 해주나

`useQuery`(읽기) / `useMutation`(쓰기) 두 훅이 강의 3의 골칫거리(캐싱·로딩/에러·재요청·중복제거·공유·저장 후 갱신)를 통째로 가져간다.

> 중요: **실제 서버 호출은 여전히 내 함수(service layer)가 한다.** TanStack Query는 그 함수를 *감싸서* 캐싱·상태관리를 입힐 뿐. 호출 자체를 대체하지 않는다.

### useQuery (읽기)

```jsx
const { data, isLoading, error } = useQuery({
  queryKey: ['cards', { bookId, kind }],   // 이 캐시의 이름표
  queryFn: () => getCards({ bookId, kind }), // 실제 호출(내 함수)
})
```

- **queryKey = 캐시의 이름표(주소).** 같은 키를 부른 화면들은 **요청 한 번, 결과 공유.** 이게 "여러 화면 동기화"를 공짜로 만드는 핵심 메커니즘.
- 반환값 `data / isLoading / error`만 보면 화면을 그릴 수 있다 — 로딩 스피너, 에러 메시지, 데이터 렌더가 한 패턴.

### 캐시의 생애 (stale / refetch) — 보충

- TanStack Query는 받은 데이터를 캐시에 두고, 기본적으로 잠시 후 "stale(낡음)"으로 표시한다.
- 사용자가 **창에 다시 포커스**하거나, 컴포넌트가 다시 마운트되면 → **백그라운드에서 조용히 refetch**해 최신으로 갱신(화면은 일단 캐시를 보여줘 빠르고, 새 데이터 오면 교체).
- 효과: "다른 기기에서 추가한 카드"가 탭 전환만 해도 슬그머니 최신화됨.

### useMutation (쓰기) + 무효화(invalidation)

```jsx
const { mutate: save } = useMutation({
  mutationFn: saveWordCard,
  onSuccess: () => queryClient.invalidateQueries({ queryKey: ['cards'] }),
})
```

- 쓰기(저장/삭제)는 `useMutation`.
- **invalidateQueries = "이 캐시는 이제 낡았다고 표시 → 다시 받아와"**. 카드를 저장하면 `['cards']` 캐시를 무효화 → 단어장 목록이 자동으로 새로고침. **수동으로 목록에 push할 필요 없음.**

### 낙관적 업데이트 (Optimistic Update) — 보충

"버튼 누르면 *즉시* 화면 반영, 서버 응답은 나중에" 패턴. 빠릿한 UX의 핵심.

```jsx
useMutation({
  mutationFn: saveWordCard,
  onMutate: async (newCard) => {
    // 1. 서버 응답 기다리지 않고 캐시를 미리 바꿈 (⊕ → ✓ 즉시)
    //    + 실패 대비해 이전 값 백업
  },
  onError: (err, vars, context) => {
    // 2. 실패하면 백업으로 롤백 (✓ → ⊕) + 토스트
  },
  onSettled: () => queryClient.invalidateQueries({ queryKey: ['cards'] }),
})
```

흐름: **즉시 성공한 척 → 실패하면 되돌림.** 99%는 성공하니 체감 속도가 빠르고, 1% 실패만 롤백. SentenceMate의 "`⊕` 누르면 즉시 `✓`, 실패 시에만 되돌림"이 정확히 이것.

### 왜 이게 "솔로 MVP에 과하지 않은가"

라이브러리 +1의 비용보다, 위 보일러플레이트(강의 3 함정)가 통째로 사라지는 이득이 훨씬 크다. 서버 상태가 많은 앱에선 **빼는 게 오히려 손해.**

## 강의 5: 클라이언트 상태 — Context vs Zustand

서버 상태를 Query가 가져가면 남는 클라 상태는 적다. 그중 **여러 화면이 공유**하는 소수만 "전역 store"가 필요(나머지는 그냥 useState).

### Context (React 내장)

- 용도: **props drilling 회피.** 멀리 떨어진 자식에게 값을 중간 단계 없이 꽂아주는 "통로".
- "라이브러리를 안 쓴 상태"가 아니라, 그 자체가 React 내장 도구.
- 단점: 값 종류가 많으면 `<Provider>`로 앱을 양파처럼 감싸야 함(provider 지옥), 리렌더 최적화가 번거로움.

### Zustand (작은 외부 라이브러리, ~1KB)

- "전역 클라 상태를 담는 작은 서랍장." store를 만들고, 컴포넌트는 **필요한 칸만 꺼내 쓰는 훅**이 됨.

```js
// store 정의
import { create } from 'zustand'
export const useSettings = create((set) => ({
  theme: 'light', fontSize: 18,
  toggleTheme: () => set((s) => ({ theme: s.theme === 'light' ? 'dark' : 'light' })),
}))
```
```jsx
// 사용 — provider 없이 그냥 import
const theme  = useSettings((s) => s.theme)        // theme 칸만 구독
const toggle = useSettings((s) => s.toggleTheme)
```

### 비교

| | Context | Zustand |
| --- | --- | --- |
| 설치 | 내장 | +1 (~1KB) |
| Provider 감싸기 | 필요(중첩됨) | 불필요 |
| "필요한 칸만 구독" | 번거로움 | 쉬움(리렌더 최적화 자동) |
| 학습 곡선 | 중간 | 낮음 |

둘 다 정답이 될 수 있다. 클라 상태가 적으면 Context도 충분. Zustand는 "더 편하고 가벼워서" 선택.

### 과설계 경계선 (제일 중요)

- **서버 데이터(카드 목록 등)를 Zustand/Context에 넣지 말 것** — 그건 Query 일. 전역 store는 "UI 설정 서랍"으로만.
- **컴포넌트 한 곳만 아는 상태(탭·시트 열림)는 그냥 useState** — store에 올리지 말 것.
- 이 절제가 상태관리를 가볍게 유지하는 핵심. "전부 전역에 넣기"가 초보의 함정.

## 강의 6: 폴더 구조 — 종류별 vs 기능별

### 두 철학

**(가) 종류별 (by type)** — 기술 역할로 묶기
```
src/
├─ components/   (모든 UI)
├─ hooks/        (모든 훅)
├─ services/     (모든 데이터 접근)
└─ stores/
```
- 장점: 단순. "훅은? → hooks/". 파일 적을 때 최고.
- 단점: 커지면 `components/`에 수십 개 쌓여 "단어장 관련"이 여러 폴더에 흩어짐.

**(나) 기능별 (by feature)** — 도메인으로 묶기
```
src/features/
├─ library/   (BookLibrary + 그 훅 + 그 조각이 한 폴더)
├─ vocab/
└─ stats/
```
- 장점: "단어장 고칠 일 = `features/vocab/` 한 폴더." → "문제 있는 화면만 건드린다"가 진짜 실현되는 건 라우터가 아니라 *이 폴더 구조* 덕분.
- 단점: 아주 작은 앱엔 과하게 느껴지고, "이 조각은 어느 feature?" 경계 고민.

### 하이브리드 (중간 규모의 정석)

> **공통(여러 화면 공유) = 종류별 / 화면 전용 = 기능별.**

```
src/
├─ app/         라우터·셸 (앱 인프라)
├─ features/    화면별 묶음 (library, reader, vocab, stats, account)
├─ services/    데이터 접근 (공통)
├─ stores/      Zustand 서랍 (공통)
├─ hooks/       여러 기능 공유 훅
├─ lib/         서드파티 래퍼 (supabase, indexeddb...)
└─ components/ui/  범용 조각 (Button, Sheet, Skeleton...)
```

규칙 셋:
1. 한 화면에서만 쓰는 것 → 그 `features/` 안.
2. 여러 화면 공유만 밖으로(services/stores/hooks/components·ui/lib).
3. 데이터 접근(`services/`)은 항상 분리 → "컴포넌트는 백엔드(DB·API)를 모른다"의 방어선.

> 핵심 통찰: **라우터는 "어느 컴포넌트를 띄울지"만 정한다. 코드가 화면별로 깔끔히 나뉘는 건 폴더 구조가 한다.** 둘은 다른 일 — 그래서 따로 정한다.

## 강의 7: 구현 시퀀싱 — 만드는 순서의 전략

설계가 "무엇을 만들지"라면, 시퀀싱은 "어떤 차례로 쌓을지". 순서를 잘못 잡으면 "몇 주 짰는데 아직 아무것도 안 돌아감"에 빠진다.

### 수평 슬라이스 vs 수직 슬라이스

**(가) 수평 (layer-by-layer) — ❌ 함정**
```
DB 전부 → service 전부 → 화면 전부
```
- 맨 마지막에 화면 붙이기 전까지 **앱이 한 번도 안 돌아감.** 설계 실수가 끝에 가서야 터지고, 통합 버그가 몰림. "90% 했는데 되는 게 없음".

**(나) 수직 (feature-by-feature) — ⭐ 권장**
```
기능 하나를 DB→service→화면까지 얇게 관통해 *돌아가는 상태*로 → 다음 기능 얹기
```
- **첫날부터 뭔가 진짜 동작.** 데이터 흐름(클라↔서버) 실수를 일찍 잡음. 동기 유지.

### 핵심 루프 먼저

앱이 "살아있다"고 말하는 최소 동작 = **핵심 루프.** 이걸 먼저 끝까지 뚫고, 나머지(부가 기능)는 그 위에 얹는다.

- SentenceMate 핵심 루프: `책 연다 → 문장 선택 → AI 물음 → 응답 → 카드 저장 → 단어장 확인`.

### 마일스톤 잡는 법

각 마일스톤이 끝나면 **"여기까지 진짜 돌아간다"**는 체크포인트가 되게 자른다.

```
M0~M1  토대(라우팅·상태·인증·DB) — 화면 비어도 골격이 삼 (불가피한 수평)
M2~M5  수직 슬라이스로 핵심 루프 완성 ← 분수령. 여기까지가 곧 MVP
M6~M8  부가 기능 얹기 (있으면 좋은 것)
```

> 원칙: **토대만 수평, 그 위는 전부 수직.** 그리고 핵심 루프 완성 지점(분수령)을 분명히 정해두면, 시간이 모자랄 때 "어디까지가 출시 가능선"인지 안다.

## 강의 8: 환경 변수와 시크릿 — 무엇을 숨기나

### 철칙

> **빌드 도구가 클라이언트에 노출하는 접두사(Vite는 `VITE_`)가 붙은 변수는 무조건 브라우저 번들에 들어간다 → 비밀 키엔 절대 그 접두사를 안 붙인다.**

브라우저로 내려간 모든 것(번들·네트워크 탭)은 사용자가 들여다볼 수 있다. "숨겼다"는 착각이 사고를 부른다.

### 공개해도 되는 것 vs 절대 안 되는 것

| 키 | 클라 노출? | 왜 |
| --- | --- | --- |
| API URL, **공개(anon) 키** | ✅ 괜찮음 | 권한 제어(RLS 등)가 따로 막아줌 — 노출 전제로 설계됨 |
| **서버 운영자 API 키** (예: LLM 키) | ❌ 절대 | 노출 = 결제 폭탄. 서버(서버리스 함수)에만 |
| **service_role / 관리자 키** | ❌ 절대 | 모든 권한 제어를 무력화하는 마스터 키 |

### 패턴

- 공개 키는 클라(`VITE_*`), 비밀 키는 **서버리스 함수의 secret**에만.
- `.gitignore`에 `.env*` (단 `.env.example`은 커밋해 "어떤 키가 필요한지" 목록만 공유).
- 출시 전 **grep으로 클라 번들·git에 비밀 키가 없는지** 확인.

> 보안의 본체는 "키를 클라에서 빼고 + 서버에서 권한 검증"이다. 이 둘만 제대로 하면 대부분 자동으로 막힌다.
