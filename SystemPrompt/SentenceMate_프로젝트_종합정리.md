## Part 1: 완료된 사항

---

### 1. 문제 정의 ✅

**근본 문제:** 영어 원서를 읽다가 모르는 문장이 나오면 몰입이 깨지고, 이 경험이 반복되면 원서 읽기 자체를 포기하게 된다.

**문제의 세 겹 구조:**
- 컨텍스트 스위칭 비용 — 원서 앱 닫기 → ChatGPT 열기 → 붙여넣기 → 답변 → 복귀
- 인지 과부하 — 단어 뜻을 찾아도 문장 전체를 재조립하는 데 에너지 소모
- 학습 휘발성 — 이해했던 표현을 며칠 뒤 다시 모름

**기존 해결책의 한계:**
- 사전 앱: 단어 뜻은 알려주지만 문장 구조/맥락 속 해석은 못 함
- ChatGPT/Gemini 직접 사용: 창 전환 비용 + 매번 원하는 포맷 안 나옴 + 응답 포맷 불일치
- 스푼리더 AI: 내장된 원서만 지원, 사용자 파일 불가
- 일반 AI PDF 리더: 영어 학습자 관점의 교사 스타일 설명에 미특화

---

### 2. 솔루션 방향 ✅

**한 줄 요약:** 사용자가 가진 영어 원서 파일을 읽으면서, 모르는 문장을 선택하면 앱 안에서 바로 AI가 "영어 메이트"처럼 설명해주는 리딩 앱.

**앱 포지셔닝:** "영어 공부 앱"이 아니라 **"영어 메이트"**. 부담감을 최대한 낮추고, 원서 읽기의 접근 장벽을 허물어주는 친구 같은 도구.

**교육적 철학 (언어학적 근거):**
- Krashen의 Comprehensible Input 가설: 이해 가능한 입력에 충분히 노출될 때 언어가 습득됨
- 이 앱의 역할: i+5 수준의 콘텐츠를 AI 스캐폴딩으로 i+1처럼 읽을 수 있게 만듦
- Affective Filter: 편하고 재미있는 상태에서 읽으면 흡수율이 높아짐. "공부"가 아닌 "즐기기"로 접근
- 핵심 균형: 접근 장벽을 낮추되, 학습자가 스스로 읽는 힘을 기르는 것을 방해하지 않는다. "목발"이 아니라 "보조 바퀴"

**타겟 사용자:**
- 1차: 본인 (dogfooding)
- 확장: TOEIC 600~900 구간, 영어 원서를 읽고 싶지만 막혀서 포기한 경험이 있는 사람

### 3. 프로젝트 목적 ✅

**1차 목적:** AI 엔지니어로서의 실전 경험 축적
- 실제 사용자 대상 앱을 Play 스토어에 출시하고 운영
- 서버 관리, API 비용 관리, 사용자 피드백 기반 개선 사이클
- 수익화 경험 (구독/과금 모델 운영 경험 포함)
- 포트폴리오: "출시 → 사용자 데이터 기반 의사결정 → 수익화 → 버전 업데이트"의 전체 사이클

**2차 목적:** 본인이 실제로 쓰고 싶은 앱 만들기

### 4. MVP 설계 원칙 ✅

**핵심 가설:** "원서 읽기 흐름을 끊지 않는 영어 메이트 스타일 AI 설명이, 사용자의 원서 읽기 지속률을 높인다."

| 원칙 | 설명 |
|------|------|
| 리더는 빌린다 | 파일 렌더링은 오픈소스 라이브러리 래핑. 직접 만들지 않음 |
| 프롬프트를 깎는다 | AI 설명의 품질/포맷/분량 최적화에 가장 많은 시간 투자 |
| 기능을 최소화한다 | 핵심 루프(파일 열기 → 문장 선택 → AI 설명)만 구현 |

---

### 5. AI 응답 구조 설계 ✅

**3단계 계층형 응답:**

| 단계 | 내용 | 성격 | 노출 |
|------|------|------|------|
| 📖 1단계 | 직독직해 (영어 어순 청크 해석 + 중급 이상 단어 뜻 병기) | "어떻게 읽는가" | 기본 노출 |
| 📝 2단계 | 자연스러운 한국어 해석 | "결국 무슨 말인가" | 탭하면 펼침 |
| 💡 3단계 | 영어식 사고방식 · 어원 · 문법 · 문화적 맥락 · 작가 의도 | "왜 이렇게 되는가" | 탭하면 펼침 |

**사용자 모드 (앱 설정에서 선택):**
- 편한 형: 1단계(직독직해) → 2단계(자연스러운 해석) → 3단계(심화 설명)
- 능동적 학습자 형: 별도 프롬프트로 구성 (미설계, 추후 진행)

**핵심 설계 결정들:**
- 기본형/상세형 구분 없이 하나로 통합. 중급 이상 단어에만 뜻 병기.
- 30초 룰: 1단계는 15초 이내 읽기 가능한 분량
- 전체 응답 1분 이내
- 난이도에 따라 3단계 분량 자동 조절 (쉬운 문장 = 짧게, 어려운 문장 = 상세하게)

### 6. ChatGPT 직접 사용 대비 차별화 ✅

| 항목 | ChatGPT 직접 사용 | SentenceMate |
|------|-------------------|--------------|
| 접근성 | 창 전환 필요 | 읽기 화면 안에서 즉시 |
| 응답 포맷 | 매번 다름 | 항상 동일한 3단계 구조 |
| 분량 | 들쭉날쭉 | 한 화면 안에 통제 |
| 맥락 | 사용자가 매번 상황 설명 | "원서 읽기 중" 맥락 상시 주입 |
| 잡담 | "Sure!", "Hope this helps!" | 바로 핵심만 |
| 어휘 설명 | 숙어 통째로 외우기식 | 핵심 이미지 접근 (give의 본질은 "상대에게 향하게 하다") |
| 한국인 특화 | 없음 | 한국어 대표 번역어와의 갭 감지 |

### 7. 시스템 프롬프트 v0.2 ✅

**완성 상태:** 초안 완성, Gemini Flash 테스트 완료

**포함된 핵심 요소:**
- 3단계 응답 구조 강제
- 직독직해: 영어 어순 그대로 청크별 해석, 중급 이상 단어 뜻 병기
- 핵심 이미지 접근법: 단어를 번역하지 않고 근본 이미지로 설명
- 한국어 대표 번역어 갭 감지: give=주다 같은 1:1 대응의 한계를 짚어줌
- 퓨샷 예시 1개 포함
- 편한 톤 강제, 잡담/마무리 멘트 차단

**테스트 결과:**
- 퓨샷 반영 후 직독직해 청크 단위 개선됨
- 어휘/문화적 맥락 설명은 양호
- 문법 설명 톤은 추가 실험 필요 (문법 용어 사용 vs 느낌 전달 — 아직 미확정)

**미결 사항:**
- 문법 설명의 최적 톤/깊이 → 앱 사용하면서 v0.3에서 개선
- 능동적 학습자 형 프롬프트 → 편한 형 확정 후 별도 설계

### 수익화 모델 (검토 필요)

> 목표: 큰 수익이 아닌, "수익화 경험"을 포트폴리오에 포함하는 것

| 모델 | 방식 | 장점 | 단점 |
|------|------|------|------|
| **프리미엄(무료+유료)** | 하루 N문장 무료, 이후 월 구독 | 가치 체감 후 전환 유도. 가장 일반적 | 무료 한도 설정이 핵심 |
| **일회성 구매** | 앱 자체를 유료로 판매 | 단순함 | 사용자 유입 장벽 높음 |
| **BYOK + 프리미엄** | 기본 무료 제공(서버 프록시) + 파워유저는 자기 API 키 | 비용 통제 가능 | BYOK 온보딩 복잡도 |

**포트폴리오 관점에서 가장 가치 있는 경험:**
- 서버 프록시 운영 (백엔드 + API 비용 모니터링)
- 프리미엄 모델 운영 (결제 연동, 사용량 추적, 전환율 분석)
- 사용자 데이터 기반 의사결정

### 리스크 & 주의사항

**기술 리스크:**
- ePub 렌더링 복잡도 → 검증된 오픈소스 라이브러리 필수
- 텍스트 선택 정확도 → 리더 라이브러리 선정의 최우선 기준
- 코드 에이전트 의존 → 상태 관리 버그는 직접 설계/디버깅 필요
- 첫 Android 개발 → 기능 수 최소화가 핵심

**시장 리스크:**
- 타겟 시장이 좁을 수 있음 ("PDF/ePub으로 영어 원서 읽는 사람")
- 킨들 AI 해설, ChatGPT 오버레이, Galaxy AI 등 대형 플레이어 위협
- 방어선 = "이 앱의 설명이 확실히 더 낫다" (프롬프트 품질)

**교육적 리스크:**
- AI 설명 의존도가 높아지면 "한국어 설명 읽기"가 될 수 있음
- 모호함에 대한 내성(추론 능력) 발달 기회 감소 가능
- 대응: 계층형 응답 구조 + 능동적 학습자 모드 + 페이지당 질문 횟수 추적
